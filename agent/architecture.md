## 关键设计决策

### 决策 1：为什么从 bash 开始？

**背景**：考虑过封装高级 API（文件操作、代码编辑等）

**决策**：保持 `bash is all you need` 理念，Agent 通过生成 bash 命令与环境交互

**理由**：
1. 通用性：bash 可以做任何事情
2. 透明性：每个操作都可观测、可回放
3. 涌现性：限制越多，涌现越难；bash 是最低限制接口

### 决策 2：模型适配层独立

**背景**：是否把 API 调用直接写在 agent.py 里？

**决策**：独立 `deepseek.py`，未来支持多模型切换

**接口设计**：
```python
# 所有模型适配器遵循相同接口
def chat(messages) -> str
```

### 决策 3：Skill 命令自动路由

**背景**：Agent 需要既能执行 bash 又能使用 skill

**决策**：`base_ops.run_command()` 自动识别命令类型
- Bash 命令：`ls -la`, `cat file.txt`
- Skill 命令：`create_file("test.txt", "hello")`

**路由逻辑**：
```
Agent 输出 -> run_command()
               │
               ├── 是 skill 格式? -> skill_dispatcher -> 执行 skill 函数
               │
               └── 否则 -> run_bash() -> 执行 shell 命令
```

**Skill 扫描机制**：
- 自动扫描 `skills/` 目录下的所有模块
- 收集各模块 `COMMANDS` 字典中的命令
- 运行时动态构建命令路由表

### 决策 3：技能系统分层

**背景**：file_ops 已经很强大，Agent 是否直接调用？

**决策**：技能是可选层，Agent 可以选择：
- 直接用 bash 操作（通用但原始）
- 调用技能（高效但依赖特定实现）

## 扩展路径

### 添加新 Agent 类型

1. 创建 `backend/<agent_name>.py`
2. 导入所需模块（base_ops、deepseek 等）
3. 实现 `main()` 入口
4. 在本文档更新状态

### 添加新技能

1. 创建 `backend/skills/<skill_name>/` 目录
2. 编写 `skill.md`（必须包含：技能介绍、命令介绍）
3. 在 `scripts/` 下实现功能
4. 在 `__init__.py` 导出接口

### 添加新模型

1. 创建 `backend/<model_name>.py`
2. 实现 `chat(messages) -> str` 接口
3. 实现 `check_api_key() -> bool` 接口
4. 添加环境变量配置说明

## 反模式警告

❌ **不要这样做**：
- 在 agent.py 里直接写死特定模型的调用
- 技能模块依赖 Agent 内部状态
- 模块间循环导入
- 把对话记录等敏感数据提交到 git

✅ **应该这样做**：
- 通过接口依赖，而非具体实现
- 技能是自包含的工具库
- 单向依赖：agent -> skills -> base_ops
- 敏感数据及时加入 .gitignore
